# トークン消費最適化ルール

## 基本原則

トークン最適化は「制約」ではなく「より多くのユースケースを実現可能にするもの」。
コスト削減・レイテンシ改善・品質向上・スケーリングの複利効果がある。

---

## 1. プロンプト最適化

### 指示は簡潔に
- 冗長な前置きや丁寧語を排除し、核心だけを伝える
- 悪い例: `Please kindly summarize the key points of the following document for me`
- 良い例: `Summarize key points`
- 効果: 最大70%のトークン削減

### システムプロンプトの構造化
- 変わらない指示（役割定義・制約・フォーマット）はプロンプトの**先頭**に配置
- 先頭に固定コンテンツを置くことでキャッシュヒット率が上がる
- 頻繁に変わる動的コンテンツは末尾に配置

### 不要な説明を省く
- AIへの説明的なコメント（「あなたはAIなので...」等）は書かない
- コードのインラインコメントは英語で1行まで
- ドキュメント: WHYが非自明な場合のみ記述

---

## 2. コンテキスト管理

### 注入ファイルの最小化
- 毎回のAPI呼び出しに付帯する情報は「核心的な状態と重要な教訓のみ」
- 目標: 26KB以下に抑える
- aidlc-docs内のファイルも読み込み量を最小化（必要なファイルのみ参照）

### コンテキストトリミング
- 長い会話では全履歴でなく「最近のメッセージ + 要約 + 重要な意思決定点」のみ保持
- 古いツール実行結果は積極的に切り捨てる
- コンテキスト窓の80%を超えたら圧縮を検討する

### オンデマンド検索の活用
- 全情報をプロンプトに詰め込まない
- 必要な部分だけをファイル読み込みで取得（ReadツールでOffset/Limit活用）
- ファイル全体でなく関連箇所のみ読む

---

## 3. モデル選択戦略

### タスク複雑度でモデルを使い分ける
| タスク種別 | 推奨モデル |
|-----------|-----------|
| 単純な情報取得・整形 | haiku (最安) |
| 標準的な実装・レビュー | sonnet (バランス) |
| 複雑な設計・推論・判断 | opus (高精度) |

### 分母最適化を優先する
- 「1回のトークン数削減」より「高コストモデルの呼び出し回数削減」の方が効果大
- 理解・判断 → opus、実行・生成 → sonnet への分担が有効
- 承認後の単純作業はhaikusに委譲する

---

## 4. 出力制御

### 出力トークンは入力より高い
- 出力トークンは入力の3〜10倍のコストがかかる
- `max_tokens`を適切に設定し、不必要な長文回答を防ぐ
- 「簡潔に回答してください」等の指示を明示する

### 構造化出力の活用
- 自由形式の長文より、JSON/リスト/表形式で応答を要求する
- フォーマット指定により冗長な説明文が減る

### 停止シーケンスの設定
- 必要な情報が得られたら生成を止める停止条件を設定する

---

## 5. キャッシング活用

### プロンプトキャッシングの最大化
- 静的なコンテンツ（システムプロンプト・ルール・ペルソナ定義）は先頭に配置
- Few-shotの例も先頭の固定部分に置く
- 注入ファイルの内容を安定させてキャッシュヒット率を上げる
- 効果: 最大50〜90%のコスト削減

### ファイル読み込みの重複排除
- 同じファイルを同一ターンで複数回読まない
- 既に読んだ内容は変数/メモリとして保持する

---

## 6. ツール管理

### ツール呼び出しの精度向上
- ツール名と説明を明確にして誤呼び出しによるリトライを防ぐ
- 不要なツールは渡さない（使わないツールのスキーマもトークンを消費する）
- エラー時は詳細で実行可能なエラーメッセージを返す（曖昧なメッセージは再試行コストを増やす）

### 並列ツール呼び出しの活用
- 独立したタスクは並列で実行してターン数を減らす
- 複数のReadやBashは1メッセージでまとめて呼び出す

---

## 7. バッチ処理

### 非緊急タスクはバッチAPIを使用
- リアルタイム不要なタスク（レポート生成・大量データ処理等）はBatch API
- Batch APIで最大50%コスト削減
- ハッカソン開発中はリアルタイムAPI優先でOK

---

## 8. RAG（検索拡張生成）

### 全文書より関連チャンクだけ渡す
- ドキュメント全体でなく検索で関連箇所のみ取得
- 効果: 全文書に比べてトークン使用量60〜80%削減
- このプロジェクトでは `grep` / `find` で必要なコードのみ読む

---

## 9. aidlc-docs読み込みの最適化

### 段階的ロード原則
- ワークフロー開始時に全ファイルを読まない
- 各フェーズで必要なルールファイルのみを読み込む
- opt-in拡張は採用決定後に初めて全文を読む

### 必要最小限のコンテキスト
- 前フェーズの成果物を参照するときは全文でなくサマリーセクションのみ読む
- audit.mdへの書き込みはEditツールで追記（全文上書き禁止）

---

## 10. 効果測定の目安

| 施策 | 期待削減率 |
|------|-----------|
| プロンプト冗長性排除 | 〜70% |
| プロンプトキャッシング | 50〜90% |
| モデルルーティング | 40〜75% |
| RAG活用 | 60〜80% |
| バッチ処理 | 50% |
| 出力長制限 | 20〜40% |

複数施策を組み合わせると全体で**月間コストを80%以上削減**することも可能。
