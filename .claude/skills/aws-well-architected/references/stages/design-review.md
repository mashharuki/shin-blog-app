# 設計フェーズ Well-Architected レビューガイド

## このガイドの目的
アーキテクチャ設計段階でWell-Architectedの原則を適用し、
実装前に問題を発見・解決するためのガイド。

**「設計段階での修正コストは、実装後の100分の1」** — 設計段階の厳格なレビューが最も費用対効果が高い。

---

## 設計レビュープロセス

### Phase 1: 設計要件の確認
設計を受け取ったら、まず以下を確認する:

```
□ ワークロードの種類（Web/API/Batch/ML/IoT等）
□ 想定トラフィック（RPS/TPS）と成長予測
□ 可用性要件（SLA: 99%, 99.9%, 99.99%等）
□ RTO/RPO 要件（障害復旧時間/データ損失許容）
□ データ分類（公開/内部/機密/規制対象）
□ 規制・コンプライアンス要件（PCI-DSS, HIPAA, SOC2等）
□ 地理的要件（リージョン、データレジデンシー）
□ チームのスキルセット
□ 予算制約
□ 既存システムとの統合要件
```

### Phase 2: アーキテクチャパターンのレビュー

#### マイクロサービス vs モノリス
```
推奨判断基準:
- チームが3-5人以下 → モノリスから開始（モジュラーモノリス）
- 独立スケーリング要件がある → マイクロサービス
- デプロイ頻度が異なるコンポーネントがある → マイクロサービス
- 技術スタックの統一が重要 → モノリス
```

#### サーバーレス vs コンテナ vs EC2
```
Lambda を選ぶ場合:
- イベント駆動処理
- 実行時間 < 15分
- ワークロードが不規則/バースト性
- コールドスタートが許容できる

Fargate/ECS を選ぶ場合:
- 長時間実行が必要
- コンテナを使いたい
- 細かいリソース制御が必要

EC2 を選ぶ場合:
- 特定ハードウェア要件（GPU等）
- OS レベルのカスタマイズが必要
- 高い予測可能なトラフィック（Reserved Instances 活用）
```

### Phase 3: 6つの柱チェックリスト（設計段階）

#### セキュリティ設計チェック（最優先）
```
🔴 CRITICAL - 設計から必ず含めること:
□ 全データフローの暗号化（転送中・保存時）
□ 認証・認可の設計（Cognito, IAM）
□ ネットワーク分離設計（VPC, サブネット）
□ シークレット管理の設計（Secrets Manager）
□ 監査ログの設計（CloudTrail, CloudWatch Logs）
□ WAF/DDoS対策（WAF, Shield）

🟠 HIGH:
□ 最小権限IAMポリシーの設計
□ データ分類と保護方針
□ インシデントレスポンス計画
```

#### 信頼性設計チェック
```
□ マルチAZ設計（SLA 99.9%以上の場合は必須）
□ サーキットブレーカーパターンの採用
□ バックアップ・DR戦略（RTO/RPO要件を満たすか）
□ Auto Scaling 設計
□ ヘルスチェックの設計
□ データバックアップ戦略（AWS Backup）
□ 依存関係の分析（サードパーティAPI等）
```

#### パフォーマンス設計チェック
```
□ キャッシュ戦略（CloudFront, ElastiCache）
□ データベースアクセスパターンとインデックス設計
□ 非同期処理の活用（SQS, SNS, EventBridge）
□ CDN設計（CloudFront）
□ 接続プーリング（RDS Proxy）
□ ページネーション設計
```

#### コスト設計チェック
```
□ サーバーレス優先設計
□ Auto Scaling でオーバープロビジョニング防止
□ データライフサイクル設計（S3 Lifecycle）
□ コストタグ戦略
□ 開発/本番環境のサイズ差異
□ データ転送コスト最小化設計
```

#### オペレーショナルエクセレンス設計チェック
```
□ IaC（CDK/CloudFormation）の採用
□ CI/CDパイプライン設計
□ モニタリング・アラート設計（CloudWatch Dashboard）
□ ログ設計（構造化ログ、集中管理）
□ デプロイ戦略（Blue/Green, Canary）
□ Runbook の策定計画
```

#### 持続可能性設計チェック
```
□ Graviton インスタンス採用の検討
□ データライフサイクル管理（TTL設定）
□ 不要データの自動削除設計
□ サーバーレスによる使用時のみ課金
```

## 典型的なアーキテクチャパターン

### パターン1: Webアプリケーション（推奨構成）
```
インターネット
    ↓
CloudFront (CDN + WAF)
    ↓
ALB (Application Load Balancer)
    ↓
ECS Fargate / Lambda (コンピュート, マルチAZ)
    ↓
├── DynamoDB / Aurora (DB, マルチAZ)
├── ElastiCache Redis (キャッシュ)
└── SQS (非同期処理)
    
補助サービス:
- Cognito (認証)
- Secrets Manager (シークレット)
- CloudWatch (モニタリング)
- CloudTrail (監査)
- AWS Backup (バックアップ)
```

### パターン2: サーバーレスAPI（推奨構成）
```
クライアント
    ↓
API Gateway (REST/HTTP/WebSocket)
    ↓
Lambda (ビジネスロジック)
    ↓
├── DynamoDB (メインDB)
├── SQS → Lambda (非同期処理)
└── EventBridge (イベント駆動)

補助:
- Cognito (認証)
- CloudFront (CDN)
- X-Ray (トレーシング)
- CloudWatch (ログ・メトリクス)
```
