# 書類審査 詳細チェックリスト

## 提出期限: 2026年5月10日（日）

---

## [A] テーマ適合性チェック（最重要）

### A-1: 「人をダメにする」コンセプトの明確性
- [ ] README.mdまたはrequirements.mdに「どのように人をダメにするか」が1〜2文で端的に説明されている
- [ ] ダメになるメカニズムが具体的（「SNS依存」のような曖昧な表現ではなく、独自の仕組みが定義されている）
- [ ] 「ダメになるユーザー」のペルソナが定義されている

### A-2: 創造性の評価
- [ ] アイデアが既存の有名サービス（Twitter, YouTube等）の単純なパロディではない
- [ ] 独自の「ダメ化」メカニズムが発明されている
- [ ] ユーモアや社会批評の要素が含まれている（加点要素）
- [ ] テーマの解釈が深く、表面的ではない

### A-3: テーマの一貫性
- [ ] 全ての提出物（README, requirements, user-stories）でテーマが一貫している
- [ ] 機能要件がテーマを実現するために設計されている

---

## [B] ビジネス意図の明確さチェック

### B-1: 価値提案
- [ ] 誰のための（ターゲットユーザー）サービスか明確か
- [ ] なぜユーザーがこのサービスを使いたいと思うか説明されているか
- [ ] 「ダメになること」がユーザーにとって魅力的に提示されているか

### B-2: requirements.mdの品質
- [ ] ファイルが存在する（`aidlc-docs/inception/requirements/requirements.md`）
- [ ] 機能要件（Functional Requirements）が番号付きリストで網羅されている
- [ ] 非機能要件（NFR）が定義されている（性能・セキュリティ・可用性等）
- [ ] 要件の優先順位（Must Have / Should Have / Could Have）が付けられている
- [ ] 各要件が曖昧でなく、テスト可能な形で書かれている

### B-3: user-stories.mdの品質（実行された場合）
- [ ] ファイルが存在する（`aidlc-docs/inception/user-stories/`）
- [ ] ペルソナが1つ以上定義されている
- [ ] User Storiesが「〜として、〜したい、なぜなら〜」形式で書かれている
- [ ] 受け入れ基準（Acceptance Criteria）が定義されている

## [C] AI-DLCワークフロー実践チェック（必須）

### C-1: ディレクトリ構造の確認
- [ ] `aidlc-docs/` ディレクトリが存在する
- [ ] `aidlc-docs/aidlc-state.md` が存在する
- [ ] `aidlc-docs/audit.md` が存在する
- [ ] `aidlc-docs/inception/` ディレクトリが存在する
- [ ] アプリケーションコードが `aidlc-docs/` 外に配置されている

### C-2: aidlc-state.mdの確認
- [ ] Inceptionフェーズのステージが記録されている
- [ ] 実行済みステージが `[x]` でチェックされている
- [ ] スキップしたステージに理由が記録されている
- [ ] 現在のフェーズステータスが正確に反映されている

### C-3: audit.mdの品質確認
- [ ] ISO 8601形式のタイムスタンプが全エントリに記録されている
- [ ] `**User Input**: "..."` フィールドにユーザーの完全な生テキストが記録されている
- [ ] 要約・言い換えが行われていない（生テキストそのもの）
- [ ] 全フェーズの承認ログが存在する
- [ ] AIの応答も記録されている
- [ ] ファイルが正しくAppendされている（過去のログが消えていない）

### C-7: 【最重要】人間によるAI監督の証拠確認（審査員フィードバック反映）
> 審査員が特に重視したポイント。audit.mdを読んで以下を確認する。

- [ ] **AIへの壁打ち・議論の記録**: ユーザーがAIの提案に対して質問・意見・反論をしている記録がある
- [ ] **修正・変更要求の記録**: ユーザーが「ここを変えてほしい」「この方向性は違う」と指示した記録がある
- [ ] **ユーザー主導の意思決定**: AIが提案し、ユーザーが最終的に選択・決定している流れが読み取れる
- [ ] **AIレビュー活用時の人間判断**: AIレビューを使っている場合、その結果に対してユーザーが議論・選別している
- [ ] **複数イテレーションの証拠**: audit.mdのタイムスタンプから複数回のサイクルが確認できる
- [ ] **全承認が即時でない**: 全ての承認がAIの出力直後に行われているだけでなく、修正・議論を経た承認がある

### C-4: Workspace Detectionの確認
- [ ] Greenfield/Brownfield判定の記録がある
- [ ] 初期状態のスキャン結果が記録されている

### C-5: Requirements Analysisの確認
- [ ] 実施された記録がaudit.mdにある
- [ ] ユーザーによる承認がaudit.mdに記録されている
- [ ] requirements.mdが生成されている

### C-6: Workflow Planningの確認
- [ ] 実行するフェーズの計画が記録されている
- [ ] ユーザーが計画を承認した記録がある

## [D] Unit分解の適切性チェック

### D-1: Unit定義の品質
- [ ] システムが複数のUnitに分解されている（または単一Unitの理由が説明されている）
- [ ] 各Unitが独立して実装・テスト可能な単位になっている
- [ ] 各UnitのスコープがDoD（Definition of Done）で定義されている

### D-2: 依存関係の明確さ
- [ ] Unit間の依存関係が図またはリストで明示されている
- [ ] 実装順序が論理的に決定されている
- [ ] 循環依存がない

### D-3: 実装計画との整合性
- [ ] 各Unitの実装計画が具体的か
- [ ] 期間や工数の見積もりが示されているか（加点要素）
