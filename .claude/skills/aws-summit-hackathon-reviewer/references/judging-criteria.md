## 【超重要】去年の優勝チームが示した「勝利の水準」

> 2025年 AWS Summit Japan 生成AIハッカソン優勝: WhiteBox お酒同好会「KanpAI」

### 審査員が下した評価（全て超えなければ勝てない）

**鶴崎修功**: **「『これもやってくれるんだ』の連続。使い倒している実感がある。」**
- 意味: 単に機能があるだけでなく、機能の深さ・多さが予想を何度も超えた

**安野氏**: **「完成度が極めて高い。1ヶ月での開発とは思えないUIの水準。」**
- 意味: UIが商用製品として出せるレベル。開発期間を言い訳にできない

**AWS山口**: **「本物の製品のような完成度。継続性におけるユーザーフィードバック設計が優れている。」**
- 意味: 「今日リリースできる」水準。かつ使い続けてもらう設計まである

**AWS松本**: **「AI活用が加点。継続的改善への言及が評価。」**
- 意味: AI-DLCを開発全工程で活用し、未来の展望まで語った

## 「人をダめにするサービス」テーマの解釈ガイド

### 優れたアイデアの特徴
- 「ダメになる」体験が具体的・ユニーク・笑えるレベルに表現されている
- ユーザーが進んで「ダメになりたい」と思えるサービス設計
- 既存の依存・中毒メカニズムを創造的に昇華している
- 社会批評・皮肉・ユーモアが組み込まれている

### 避けるべきアイデア
- 倫理的に問題のある「ダメ化」（詐欺・依存症の深刻化等）
- 月並みな「SNS依存」「ゲーム廃人」系（差別化不足）
- アイデアが浅く、「ダメになる」要素が表面的

## 技術要件チェックリスト（決勝版）

### 必須技術要件
- [ ] AWSにデプロイされ公開URLが存在する
- [ ] 公開GitHubリポジトリ（プライベートは失格）
- [ ] AWS Builder IDを保有するチームメンバー
- [ ] AI-DLCワークフローの全フェーズ実践証拠（aidlc-docs/）

### 推奨技術スタック（AWS）
AWS Well-Architectedに準拠した構成が評価される：
- **コンピューティング**: Lambda, ECS, EC2, App Runner
- **ストレージ**: S3, DynamoDB, RDS, Aurora Serverless
- **API**: API Gateway, AppSync
- **AI/ML**: Bedrock, SageMaker, Rekognition, Comprehend
- **認証**: Cognito
- **インフラ**: CDK, CloudFormation, SAM
- **モニタリング**: CloudWatch, X-Ray
- **CI/CD**: CodePipeline, CodeBuild, GitHub Actions

### 加点要素（福井さん・塚田さんが評価）
- [ ] IaC（CDK/CloudFormation）でインフラが定義されている
- [ ] CI/CDパイプラインが構築されている
- [ ] X-Ray等のトレーシングが導入されている
- [ ] DDD/クリーンアーキテクチャに基づいた設計

## よくある失格・減点パターン

1. **GitHubがprivate** → 即失格
2. **aidlc-docs/なし** → AI-DLC実践なしとみなされ大幅減点（福井さんが即指摘）
3. **audit.mdの不正確な記録** → ワークフロー実践の証拠として認められない（福井さんが精読）
4. **テーマとの乖離** → 「人をダめにする」に程遠い場合は落選（伊沢さんが指摘）
5. **AWSを使用していない** → 技術要件未達で失格（塚田さんが確認）
6. **デモが動作しない** → 決勝での致命的欠陥（全員が評価）
7. **AIへの意思決定の丸投げ** → audit.mdを見てユーザーの議論・修正要求がない場合は大幅減点（福井さんが最重視）
8. **AIレビュー結果の鵜呑み** → AIレビューを使った場合でも人間が判断した記録がないと減点
9. **セキュリティ設定の不備** → APIキーのハードコード等（塚田さん・福井さんが指摘）
10. **NFR未定義** → 性能・セキュリティの要件が「なんとなく」の場合（福井さんが指摘）
