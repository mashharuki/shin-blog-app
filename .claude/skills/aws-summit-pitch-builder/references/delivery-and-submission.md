# デリバリー & 提出ガイド

ステージ登壇の準備中、またはデモ動画・書類審査の最終チェックをする時に読む。

---

## ライブステージ ピッチ

### 本番前チェックリスト
- ステージを確認: プレゼンターモニターはあるか？マイクタイプ（ハンドヘルド、ラベル、ヘッドセット、演台）？クリッカー？オンラインプラットフォームの設定？
- 全埋め込み動画をライブ前にテストする。外部動画はステージで壊れる。動画が失敗しても: 反応しない、謝らない、話し続ける。
- 早めにステージエリアを歩く。イベント開始前に自分がプレゼンしている様子をビジュアライズする。

### マイクコントロール
- 最適: 口から10cm、わずかに角度をつける
- マイク初心者: あごにつけて固定する。動くマイク = 音量が上下する = 聴衆が離れる

### ボディと動き
- 2m²の箱を想像する。それがあなたのパフォーマンスゾーンだ。アンカーを保つ。
- 意図的に動く: 2〜3歩、止まる、キーラインを届ける、戻る。意図的な動き = 自信。神経質なペーシング = 気を散らす。
- スライドを読むために後ろを向かない。ピッチを暗記するかプレゼンターモニターを使う。

### アイコンタクト
- 右、中央、左、目線より少し上の3つの想像上の点を作る
- 話しながらスキャンする。全員があなたが見ていると感じる

### 声とパフォーマンス
- 沈黙を使う。最強のラインの後にポーズを入れる。着地させる。
- 表情を使う: 問題ではよりエネルギーを、ソリューションでは本物の確信を
- ピッチはパフォーマンスだ。ステージに立つ人物はチーム全員の成果を代表するキャラクターだ

### 何かが壊れた時
- 動画が失敗 → 話し続ける。デモで見せたはずのものを言葉で説明する。進む。謝らない。
- スライドが進まない → 落ち着いてもう一度クリック、または次へスキップ。「申し訳ありません、技術的なトラブルで」とは言わない。
- 内容を忘れた → 遷移フレーズで橋渡しする（「そして、そこから話は...」）。凍り付いて謝らない。
- マイクが死んだ → 大きく話す、続ける。会場は対応する。

暗黙のルール: 審査員は完璧なピッチより確信に満ちた姿勢を尊重する。

## AWS Summit ハッカソン 書類審査 提出

書類審査はライブピッチとは異なる。審査員は自分のペースで確認する。
デモ動画の品質が特に重要で、明確さがパフォーマンスに勝る。

### 1. 提出物全体の時間を計る
デッキ + 動画 + 説明文の全体を、知らない人に試してもらう。
7分以上かかるなら削る。理想は審査員が4分で「何を作ったか、なぜ重要か」を理解できること。

### 2. 自分を録画して見返す
ほとんどのビルダーは自分のピッチを見ない。やれ。
フィラーワード、信頼破壊言語、ペース問題、デッドスライドを5分で見つけられる。

### 3. デモ動画は単独で成立しなければならない
一部の審査員は何も読む前にデモを見る。コンテキストなしで何が起きているか理解できなければ、ピッチに注意を向けない。
BGMなし、外部ナレーションなし、急いでスクロールしない。画面を録画し、自分でナレーションし、Wowモーメントを最初に見せる。

### 4. 使い方の前に何ができるかを説明する
ユースケースから先に。専門用語は少量なら許容されるが、ユースケースをシンプルに説明できる前に技術的になりすぎると審査員が離れる。

### 5. AWS活用の証拠を見せる
書類審査でも以下を含めることで評価が上がる:
- アーキテクチャ図（シンプルで読めるもの）
- 使用したAWSサービスと選定理由のリスト
- 実測パフォーマンス指標（Lambdaレイテンシ、DynamoDB読み書き速度など）

## デモ動画 専用ガイド

### 黄金ルール
- **最初の15秒にWowモーメント**: 最も印象的な機能を最初に持ってくる
- **シンプルなユーザーフロー**: 1つのユースケースを端から端まで見せる（全機能を詰め込まない）
- **ライブ環境**: 開発環境でなく、実際のAWS環境でのデモを優先
- **自分でナレーション**: 画面が動く間、リアルタイムで説明を加える
- **2〜3分以内**: それ以上は長すぎる

### 動画の構成（2分30秒の場合）
```
0:00-0:15   Wowモーメント（最もインパクトのある機能）
0:15-0:45   問題が起きる状況 → プロダクトが解決する
0:45-1:45   メインユーザーフロー（ステップバイステップ）
1:45-2:10   結果・インパクト（数字で）
2:10-2:30   CTA（デモURL、GitHub、連絡先）
```

### 避けること
- 開発者向けコードウォークスルー（ユースケースを先に）
- AWS管理コンソールを見せすぎる（エンドユーザー体験にフォーカス）
- 早すぎる操作（審査員が追えない）
- テキストウォールのスライドから始まるデモ

## 書類審査 提出チェックリスト

提出前に確認:

- [ ] デッキはシングルPDF（動画は埋め込みまたは別ファイル）
- [ ] スタンドアロンのデモ動画を別途アップロード（審査員が最初に見るかもしれない）
- [ ] 書面説明がデッキの内容と矛盾していない
- [ ] プロダクトへのライブリンク（本番環境推奨）
- [ ] GitHubリンク（クリーンなREADME、審査員はコード品質をここで見る）
- [ ] 使用したAWSサービスのリストと選定理由が含まれている
- [ ] アーキテクチャ図が含まれている
- [ ] 全体のレビュー時間を計った: 知らない人が7分以内で理解できる
- [ ] デモ動画が単独で成立する（コンテキストなしで理解できる）

## 最後のルール

書類審査を通過したチームは提出期限前に全てを完成させ、録画前に20回以上ピッチを練習した。
当日に慌てて録画したチームは勝てない。
ハッカソンの採択率は低い。提出物を最も重要な成果物として扱え。
