# コーチングレベルのブラッシュアップ & デザインベストプラクティス

Step 5: ポリッシュパスで適用する。1対1コーチングセッションで適用するルール群。
普通のデッキを勝てるデッキに変える最後の20%がここにある。

---

## スライドレベルのポリッシュ

### イントロスライド: 1秒テスト
見知らぬ人があなたのイントロスライドから1秒で何をするか理解できなければ、タグラインを書き直す。
正確なカテゴリ定義は汎用スローガンに勝る。

> ❌ 「AIで業務を革新するプラットフォーム」
> ✅ 「サボることを科学する。AIがあなたの代わりにToDoを整理します。」

### スタートアップ名とプロダクト名が違う場合
聴衆が痛みを理解するまでプロダクト名を紹介しない。スタートアップ名をイントロに。
プロダクト名はソリューションスライドで登場させる。そこで初めて頭に刺さる。

### カテゴリを正確な言葉で述べる
「タスク管理AIアシスタント」は「業務効率化プラットフォーム」より刺さる。

### イントロスライドの背景画像
価値提案をプロダクトと感情的につなぐビジュアルを追加する。
あなたが一言話す前から、画像が半分売っている。

## 数字とデータ

### MRR vs 累計収益
「月額MRR 50万円」は「3年間の累計売上1億円」より強い。
審査員は年次繰り返し単位で考える。

### ヒーロー指標を早めに提示する
「Slackユーザーは毎日3時間タスク管理に費やしている」を最初の一言にする。
大きな数字が注目を引く。内訳が信頼性を築く。

### スライドに主要データを大きく
MRR、アクティブユーザー数、処理件数は、そのスライドで最も大きいテキストにする。
聴衆が目を細めるなら、データポイントは無駄だ。最も印象的な数字を最大のフォントで。

### 正確な数字を使う（「以上」は禁止）
「2億358万円」は「2億円以上」より強い。精度 = 自信。

### 技術を知られているプロダクトと比較する
「Apple Watchの心拍数モニタリングより40%精確」は即座に理解される。「高度なバイオメトリクス精度」は伝わらない。
AWS Summit 向け: 「ChatGPTでできなかったSlack連携をBedrockで実現した」のような対比。

## ナレーションとフロー

### 全スライドに遷移文を書く
「だからこそ私たちは[プロダクト]を作りました」が問題→ソリューションをつなぐ。
「では実際どう収益化するか？」がプロダクト→ビジネスモデルをつなぐ。
**橋渡しが書けないなら、スライドの順序が間違っている。**

### アニメーションで機能を1つずつ表示
全機能を一度に見せない。聴衆が先読みして聞くのをやめる。
クリック→機能1→説明→クリック→機能2。ナレーティブをコントロールする。

### ポジティブなデモシーンを選ぶ
AFTER状態を見せる。ユーザーが笑顔になる、タスクが完了する、時間が節約される。
喜びは恐怖より売れる。

### チームスライドに創業ストーリーを加える
「[ハッカソン]で出会い、両方とも[問題]にキレたので作った」は
どんな資格リストよりも「なぜこのチームか」に答える力がある。

### インパクトある締め言葉で終わる
「ありがとうございました」だけでなく:
「私たちのユーザーは毎日2時間を取り戻しています。今月だけで1万時間が節約されました。」
または「先週[X]件の処理を完了しました。来月が楽しみです。」

## AWS固有のポリッシュ

### AWSアーキテクチャスライドを具体的に
サービス名を列挙するだけでなく、「なぜ」を添える:
- 「Lambdaを選んだ理由: コールドスタート時でも15秒以内の応答保証」
- 「DynamoDBを選んだ理由: シングルテーブル設計でレイテンシ一桁ms」
- 「Bedrockを選んだ理由: IAMベースの認証でAPIキー管理不要」

### コスト効率を数字で示す
AWS Summit の審査員はコスト意識を評価する:
- 「月額[X]円で[Y]件の処理を実現（EC2常時稼働の場合の10分の1）」
- 「Graviton3採用でLambda実行コストを20%削減」

### セキュリティのアピールポイントを1つ含める
AWS Well-Architected に沿った設計は審査員の信頼を高める:
- 「全シークレットはSecrets Managerで管理、ハードコードゼロ」
- 「IAMは最小権限原則、全リソースにタグ付け済み」
- 「CDK Nagで100のベストプラクティスを自動チェック」

## デモと動画

### 動画ファイルをデッキに直接埋め込む
Wi-FiはステージでDropする。外部Loom/YouTube/Vimeoリンクは壊れる。
デモは30秒以内、「Wowモーメント」は最初の5秒に。

### デモに自分で声をつける
事前収録のナレーションや BGMは使わない。デモが流れる間ライブで声をつける。
聴衆との接続を保ち、ペースを調整できる。

### AWSのライブ環境で動かすことを見せる
開発環境でなく、本番またはステージングのAWS環境でのデモを優先。
CloudWatchのメトリクスが動いている、Lambda実行ログが出ている等は審査員に刺さる。

## スクリプトと練習

### スクリプトで時間を半分に削れる
ぶっつけ本番の創業者は常に2倍超過する。スクリプトを使って練習する。
タイマーを使って全練習ランで計測する。

### Loom（またはスクリーンレコーダー）で練習する
自分を録画し、最低3回見返す。本番前に。
フィラーワード、信頼破壊言語、ペース問題、デッドスライドを5分で見つけられる。
気づかないままだと致命的だ。

### 最低20〜40回のリハーサル
当日だけでやって勝ったチームはいない。自然になればなるほど、声、感情、体、聴衆への働きかけで遊べる。

### Q&A対策のアペンディックススライドを準備する
競合差別化、詳細アーキテクチャ、コスト計算、財務予測、技術仕様は本編に入れず、
Q&Aで引き出せるように準備する。「いい質問ですね、データを見せます」は「それは本編で話しました」より圧倒的に強い。

## デザインベストプラクティス

スライドデザインのアドバイス時:

**やること**:
- クリーンなレイアウト、太いタイポグラフィ、一貫したカラーパレット
- 余白: スライドを呼吸させる
- スライドあたり1つのビジュアル、大きな主要数字
- リズムとインパクトのための大きなタグライン遷移
- 主要な所見を見出しに書いた1スライド1チャート
- デモ動画を直接埋め込む

**やらないこと**:
- 小さいフォントのごちゃごちゃしたチャート
- 誰も読めない組織図
- 1スライドに複数の競合するビジュアル
- テキストの壁
- グレーや低コントラストのテキスト（特にオンライン/投影では見えない）
- 純粋な黒背景: ダークグレーやダークブルーの方がプロっぽく目に優しい
- スライド間で混在するビジュアルスタイル
- タイポグラフィのミス。ピッチデッキの誤字1つが信頼を壊す

**パートナーロゴのルール**: 表示するなら各ロゴについて何か具体的なことを言う。文脈のないロゴは装飾だ。

**ブランドテンプレート**: 一貫させる。レイアウト混在のデッキはプロらしくない。
1テンプレート、1フォントシステム、1カラーパレットを全体で使う。

## 避けるべき構造的な罠

### 「プラットフォームのビジネスモデル探し中」の罠
内部セールスドキュメントや技術ショーケースのように読めるデッキ。
兆候: 問題・ソリューション・市場より先に複数の「技術ショーケース」スライド、ジャーゴン重症、技術詳細詰め込み。
**修正**: 商業的な証拠をリードに置く。収益、ユーザー、トラクションを前に持ってきて信頼を積んでから技術に入る。

### 「複数のストーリー」の罠
コンシューマーアプリ + B2B API + 別機能のデッキが3つのストーリーを同時に語ろうとする。結果: どれも刺さらない。
**修正**: 最も強い1つのストーリーラインを選び、他はその支持証拠にする。

### 「なぜ今か」のギャップ
強い技術と市場を提示しながら、なぜ投資家や審査員が今動くべきかを答えない。
AWS固有の追い風（Bedrock新モデル、生成AIの普及、特定業界の規制変化）は金だが埋もれていることが多い。
**修正**: 問題またはソリューションスライドに「なぜ今か」を格上げする。2〜3の決定的な力: エコシステムの変曲点、規制の転換、競合空白。

### ライブピッチでの章立てと目次
削除する。フローは連続的なナレーティブでなければならない。報告書のように章立てしない。

### 文脈なしの収益予測
市場・GTM戦略・価格・現在のトラクションを理解していない審査員には、どんな収益曲線も信頼できない。
正当化なしの収益グラフは恣意的に見える。

### ロードマップに測定可能なターゲットがない
各マイルストーンに数字が必要: 「Q2に月額MRR 50万円」「Q4に月間アクティブユーザー1万人」。
曖昧なフェーズ（「フェーズ2: スケール」）は何も意味しない。
