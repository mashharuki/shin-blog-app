# 設計フェーズ 支援ガイド
## Application Design + Units Generation

---

## このフェーズの目的

要件を**実装可能なアーキテクチャと作業単位**に変換する。
「何を作るか」→「どう作るか」への橋渡し。

---

## Application Design の進め方

### コンポーネント設計の原則

- **単一責任**: 各コンポーネントは1つの責務のみを持つ
- **高凝集・低結合**: コンポーネント内は密接に関連、コンポーネント間は疎結合
- **依存関係の方向**: 上位層が下位層に依存（逆は避ける）

### components.md テンプレート

```markdown
# コンポーネント設計

## [コンポーネント名]

**責務**: [何をするコンポーネントか1文で]

**入力**:
- [入力データ・イベント]

**出力**:
- [出力データ・イベント]

**依存コンポーネント**:
- [依存先のコンポーネント名]

**AWSサービスマッピング**:
- [対応するAWSサービス]
```

### services.md テンプレート

```markdown
# サービス定義

## [サービス名]

**種別**: [REST API / GraphQL / イベント駆動 / バッチ]
**エンドポイント**: [URL / ARN / トピック名]
**ポート**: [ポート番号（ローカル開発時）]

**提供インタフェース**:
| メソッド | パス | 説明 |
|---------|------|------|
| GET | /resource | リソース一覧取得 |
| POST | /resource | リソース作成 |

**使用するコンポーネント**:
- [コンポーネント名]
```

### AWSアーキテクチャ選定ガイド

設計時に以下の観点でAWSサービスを選定する。
詳細設計は `aws-specialist` エージェントに委譲すること:

**コンピューティング選定フロー**:
```
イベント駆動・短時間処理 → Lambda
コンテナ・長時間処理    → Fargate / ECS
常時起動・高カスタマイズ → EC2
フルマネージドコンテナ  → App Runner
```

**データストア選定フロー**:
```
柔軟なスキーマ・スケーラブル → DynamoDB
リレーショナル・トランザクション → Aurora Serverless
グローバル分散クエリ → Aurora DSQL
検索・集計 → OpenSearch
キャッシュ → ElastiCache
```

## Units Generation の進め方

### 良いUnit of Workの条件

1. **独立して実装・テスト可能**: 他のUnitが完了していなくても開発できる
2. **明確なスコープ**: 何が含まれ何が含まれないかが明確
3. **適切な粒度**: 1〜3日で完了できる規模（ハッカソンはより小さく）
4. **明確なDoD（Definition of Done）**: 完了基準が客観的に判断できる

### unit-of-work.md テンプレート

```markdown
# Unit of Work 定義

## Unit一覧

| ID | Unit名 | 優先度 | 見積工数 | 依存Unit |
|----|--------|--------|---------|---------|
| U-01 | [名前] | 高 | 4h | なし |
| U-02 | [名前] | 高 | 6h | U-01 |
| U-03 | [名前] | 中 | 8h | U-01 |

---

## U-01: [Unit名]

**スコープ**:
- 含まれるもの: [機能・ファイル・APIエンドポイント等]
- 含まれないもの: [明示的に除外する範囲]

**技術スタック**:
- 言語: [TypeScript / Python等]
- フレームワーク: [Hono / Fastify等]
- AWSサービス: [Lambda / DynamoDB等]

**Definition of Done**:
- [ ] [テスト可能な完了条件1]
- [ ] [テスト可能な完了条件2]
- [ ] ユニットテストのカバレッジ 80%以上
- [ ] CDKでデプロイ可能

**インタフェース定義**:
[このUnitが提供・消費するAPI・イベント]
```

### ハッカソン向け Unit 分解の実践例

書類審査（5/10締切）を考慮した分解戦略:

```
Inception完了（書類審査提出）→ 以下のUnit構成例

U-01: 認証・基盤インフラ（最初に実装）
  - AWS Cognito + API Gateway + DynamoDB テーブル定義
  - CDKスタック基本構成

U-02: コアAPI（「人をダメにする」中核機能）
  - ビジネスロジックの核心部分
  - Lambda関数 + DynamoDB操作

U-03: フロントエンド
  - React/Next.js + Amplify
  - コアAPIとの連携

U-04: AI/ML機能（あれば）
  - Bedrock統合
  - 生成AIによる「ダメ化」機能強化
```

### unit-of-work-dependency.md テンプレート

```markdown
# Unit依存関係マトリクス

## 依存関係図

\`\`\`mermaid
graph TD
    U01[U-01: 認証・基盤] --> U02[U-02: コアAPI]
    U01 --> U03[U-03: フロントエンド]
    U02 --> U03
    U02 --> U04[U-04: AI機能]
\`\`\`

## 実装推奨順序

1. U-01（依存なし）
2. U-02（U-01完了後）
3. U-03 + U-04（並行可能、U-02完了後）
```

## 設計レビューチェックリスト

設計完了後に確認する:

**アーキテクチャ**
- [ ] 全ての機能要件がコンポーネントに割り当てられているか
- [ ] 単一障害点（SPOF）がないか
- [ ] AWSのマルチAZ・マネージドサービス活用ができているか

**Unit分解**
- [ ] 各UnitにDoDが定義されているか
- [ ] Unit間の依存関係が循環していないか
- [ ] 最初に実装すべきUnitが明確か（MVPへの最短経路）

**セキュリティ（設計時確認）**
- [ ] 認証・認可の設計が含まれているか
- [ ] シークレット管理方針が定義されているか
- [ ] 最小権限原則でIAMを設計できているか
