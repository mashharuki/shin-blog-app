# プレゼン・コーチング

## 概要

このスキルでは3フェーズでプレゼン設計を支援する：

1. **Discovery（目的・聴衆・メッセージの明確化）**
2. **Framework Selection（ナラティブ構造の選定）**
3. **Slide Design（スライド設計とフィードバック）**

---

## Phase 1: Discovery

プレゼンの設計を始める前に、必ず以下を確認する。詳細な分析フレームワークは `references/discovery-framework.md` を参照。

### 最低限ヒアリングする4点

1. **目的**: 理解促進 / 承認獲得 / 行動喚起 / 状況共有 のどれか
2. **聴衆**: 役職・知識レベル・主な関心事・感情的スタート地点
3. **コアメッセージ**: 「このプレゼンで1つだけ覚えてもらうこと」を1文で
4. **制約**: 発表時間・スライド枚数目安・形式（登壇/会議/オンライン/配布）

ユーザーがこれらを曖昧にしか答えない場合は、`references/discovery-framework.md` の選択肢リストを提示して絞り込む。

ヒアリング完了後、以下のフォーマットで「プレゼン設計シート」をユーザーに確認させる：

```
──────────────────────────────
プレゼン設計シート
──────────────────────────────
【目的】_______________
【聴衆】役職: ___ / 知識: ___ / 関心: ___ / 感情: ___
【コアメッセージ】_______________
【制約】___ 分 / ___ 枚 / 形式: ___
【既存コンテンツ】□ データあり  □ 事例あり  □ ゼロから
──────────────────────────────
```

## Phase 2: Framework Selection

設計シートをもとに最適なナラティブ構造を推薦する。詳細な各フレームワークのスライド構成は `references/story-frameworks.md` を参照。

### 選択ロジック

| 目的 | 推奨フレームワーク |
|---|---|
| 経営承認・新規提案・投資判断 | PREP法 |
| 業務改善・システム導入・コスト削減 | 問題解決型 |
| 月次報告・進捗・調査結果共有 | SDS法 |
| 組織変革・DX推進・戦略転換 | AS-IS → TO-BE型 |
| キックオフ・ビジョン発表・採用説明会 | Why → What → How型 |
| ツール選定・ベンダー比較・戦略オプション | 比較分析型 |

推薦の際は：
- 推薦フレームワーク名と理由を1文で説明する
- スライド構成の全体像（枚数・各スライドのタイトル案）を示す
- 必要に応じてユーザーに選択肢を提示する

---

## Phase 3: Slide Design

スライド構成が決まったら、各スライドの内容を設計する。

### スライドタイトルのルール

タイトルは「トピック名」ではなく「メッセージ（結論）」で書く：
- NG: 「現状の課題」 → OK: 「手作業が月50時間の無駄を生んでいる」
- NG: 「提案内容」 → OK: 「自動化で工数を80%削減できる」

### 情報量チェック（1スライドあたり）

- 箇条書き: 最大5点、1点12語以内
- 文字数目安: 150文字以内（タイトル除く）
- 画像・図: 1点が原則

詰め込みになっているスライドは分割を推奨。詳細なスライドタイプと適切なコンテンツの対応表は `references/story-frameworks.md` を参照。

---

## 既存スライドのレビューモード

ユーザーが既存のスライドや資料を持ち込んだ場合：

1. **Discovery確認**: 目的・聴衆・コアメッセージが明確か確認
2. **構造診断**: 現在の構成がどのフレームワークに近いか分析、ずれがあれば指摘
3. **スライド単位レビュー**: タイトルがメッセージ型か・情報量・視覚的一貫性
4. **改善提案**: 優先度順に箇条書きで提示（最大5点に絞る）

---

## 出力スタイル

- 提案は具体的に。「改善しましょう」ではなく「このタイトルを〇〇に変更する」
- ユーザーが多忙な場合は要点を先出し（PREP法を自分でも使う）
- スライド構成を提示するときは番号付きリスト形式で全体像を示す
